# План: production-ready оркестратор

Документ описывает целевую архитектуру оркестратора agentsec и план работ.
Часть A (ядро/граф) реализуется в этом PR; части B и C интегрируются по
контрактам ниже.

## 1. Проблема текущего MVP

Оркестратор — это `create_react_agent`: LLM сам решает, вызывать ли
специалиста. На реальном прогоне это привело к пропуску специалиста
`authnz` — класс уязвимостей просто не был проверен, и пользователь об
этом не узнал. Сопутствующее:

- управление потоком недетерминировано — нет гарантии покрытия;
- специалисты вызываются последовательно (медленно);
- упавший/пустой специалист («отчёт 0 симв.») молча теряется;
- вывод — markdown, который парсится best-effort; нет машинной схемы;
- нет валидации находок → false positives уходят в отчёт;
- нет вердикта quality gate, хотя ТЗ требует «уточнить, можно ли
  отдавать проект на сборку».

## 2. Целевая архитектура — детерминированный StateGraph

LLM решает *что найдено*, граф решает *что запускается*. Управление
потоком — фиксированный `StateGraph`, не решение модели.

```
START → intake → (clarify?) → recon ─┬─→ specialist:injection ─┐
                                     ├─→ specialist:secrets   ─┤
                                     └─→ specialist:authnz    ─┘
                                                               │
                          consolidate ←──────────────────────┘
                               │
                          validate → gate → report → END
```

- **intake** — интерпретирует задачу на естественном языке (в т.ч.
  нечёткую); решает, нужно ли уточнение у пользователя.
- **clarify** — задаёт вопрос через `ask_user` (только в интерактивном
  режиме); иначе пропускается.
- **recon** — детерминированно запускает разведку (миньоны explore/docs)
  и, опционально, сканеры. Управление не зависит от LLM.
- **3 специалиста** — узлы графа, исполняются параллельно (рёбра из
  recon в общий superstep). Запускаются ВСЕГДА — отсюда гарантия
  покрытия и отсутствие бага с пропуском authnz.
- **consolidate** — дедуплицирует находки по fingerprint, мёржит
  `found_by`, нумерует (F-001…), сортирует по severity.
- **validate** — агент-«скептик», evidence-only контекст; режет FP,
  ставит status и CVSS.
- **gate** — детерминированный вердикт (PASS / REVIEW / FAIL) по порогам
  severity; в интерактиве спрашивает у пользователя финальное решение.
- **report** — рендер markdown + структурированный JSON через
  `reporting.save_report`.

Каждый узел изолирован: исключение/пустой результат специалиста не
роняет граф — узел ретраится, затем помечается coverage-gap.

## 3. Часть A — объём этого PR

## 4. Контракты с B и C

- **A ← B.** B отдаёт специалистов как функции `(task) → markdown` (или
  позже `→ list[Finding]`). До готовности B: специалисты в A работают
  через текущие ReAct-агенты, их markdown парсится в `Finding`
  переиспользуемым парсером `schema.parse_findings_markdown`. B владеет
  `validator.py` (контент), `prompts.py` (промпты специалистов), может
  заменить парсер на `with_structured_output`.
- **A ← C.** C отдаёт миньоны/сканеры как tools и `reporting.save_report`
  как функцию `(findings, verdict, coverage) → файл`. Эти контракты уже
  есть в репозитории.
- **Стабы.** Узел-специалист с включённым `CONFIG.stub_specialists`
  возвращает `[]` — A собирается и тестируется без B/C.

## 5. Решения по предложениям части C (внешние репозитории)

Принято / отклонено — с обоснованием:

- **Skill-packs (формат `SKILL.md` + `reference/`) — ПРИНЯТО частично.**
  Берём *формат*, не код: лёгкий `agentsec/knowledge/` с краткими
  чеклистами по классам, написанными нами и подключаемыми в промпты
  специалистов. Импортировать 26 чужих markdown-файлов — балласт и
  вопрос провенанса. Владелец — B (улучшение детекта).
- **Blind validator (evidence-only) — ПРИНЯТО.** Узел `validate`
  получает находку без severity/confidence специалиста и переоценивает
  независимо. Снижает confirmation bias. Реализуется в A (каркас) + B
  (контент промпта).
- **Evidence-package как контракт отчёта — ПРИНЯТО облегчённо.** Поле
  `Finding.evidence` несёт сырые выдержки и code-references; `reporting`
  может опционально разворачивать пер-finding каталог. Полная структура
  `findings/F-001/evidence/...` — roadmap, не блокер.
- **ast-grep / tree-sitter / trivy / trufflehog — ОТЛОЖЕНО.** trufflehog
  дублирует gitleaks, trivy fs дублирует osv-scanner. Паттерн обёртки
  сканера тривиально расширяем — добавим при доказанной нехватке
  покрытия. Порядок «сканеры → LLM» уже зашит в recon.
- **MCP-серверы для nmap/nuclei/httpx/ffuf/katana/arjun — ОТКЛОНЕНО.**
  Это DAST/сетевые инструменты для живой цели. Наша система по ТЗ —
  quality gate по *исходному коду* (доступ к кодовой базе, статический
  анализ). Запущенной цели нет. Вне скоупа.
- **Anti-loop подсистема (VulnClaw) — ОТКЛОНЕНО как подсистема.**
  Детерминированный граф уже устраняет петли управления. Внутри
  ReAct-специалиста петлю ограничивает `recursion_limit` + ретрай узла с
  пометкой coverage-gap. Отдельные `anti_loop.py`/`loop_controller.py`
  не нужны — структура графа и есть anti-loop.
- **Eval: OWASP Benchmark / DVWA / VulHub / Cybench — ОТЛОЖЕНО.**
  OWASP Benchmark Java-центричный и огромный; DVWA/VulHub — docker-цели
  для DAST. Вместо этого: расширить `examples/` несколькими маленькими
  намеренно-уязвимыми проектами (полиглот) с курируемым `expected.json`,
  и перевести eval со substring-матча на матч по структурным `Finding`
  (CWE + файл) → precision/recall.
