# План: большие репозитории и контекст по кодовой базе

Документ — план дальнейшей работы. Цель: система должна анализировать
большие репозитории, имея устойчивый **контекст** по ним (карту кода,
attack surface, граф вызовов), а не переоткрывать репозиторий с нуля
на каждом прогоне.

## 1. Диагноз: почему сейчас не тянет большие репо

У системы **нет модели репозитория**. Каждый агент (recon, специалисты,
валидатор) — это ReAct-цикл, который заново исследует код через
`grep`/`glob`/`read_file` в пределах своего лимита шагов. Подтверждено
прогонами:

- `examples/vulnerable_app` (1 файл) — 4–8 мин;
- `./agentsec` (19 файлов) — специалист `authnz` отработал **602 с**,
  суммарно > 11 мин; большой репо ушёл бы в таймаут целиком.

Конкретные потолки в коде:

- `read_file` обрезает файл на `read_file_max_chars=20000` — большой
  файл виден наполовину, окна/диапазона нет ([filesystem.py:61](../agentsec/tools/filesystem.py#L61));
- `glob_files` молча отдаёт первые 200 путей ([filesystem.py:146](../agentsec/tools/filesystem.py#L146));
- `grep` молча останавливается на 300 совпадениях ([filesystem.py:181](../agentsec/tools/filesystem.py#L181));
- `build_callgraph` ограничен 150 файлами / 500 рёбрами и отдаёт плоский
  список рёбер без достижимости ([callgraph.py:83](../agentsec/tools/callgraph.py#L83));
- recon отдаёт специалистам **свободный текст** разведки — на большом
  репо он либо поверхностный, либо не влезает в контекст;
- специалисты не знают, куда смотреть в первую очередь → блуждают →
  упираются в `specialist_timeout_sec=240` → coverage-gap → класс
  уязвимостей фактически не проверен;
- `coverage` со статусом `done` на большом репо вводит в заблуждение:
  «проверено» не значит «просмотрены все релевантные файлы».

Вывод: нужны две опоры — **(1) устойчивый контекст-индекс кодовой базы**
и **(2) масштабируемый прогон** (триаж, шардирование, бюджеты, diff-режим).

## 2. Что уже есть и работает

- Детерминированный `StateGraph`: специалисты не пропускаются.
- Параллельный fan-out специалистов и валидатора.
- Структурная схема `Finding`, дедупликация, quality gate с exit-code.
- Сессии + SQLite-чекпоинтер: долгий скан переживает паузы и рестарт —
  **это уже полезно для больших репо** (долгий прогон не теряется).
- Детерминистические сканеры (semgrep/gitleaks/osv) как pre-pass.
- Жёсткий таймаут специалиста — рантайм ограничен сверху.
- Миньоны `resolve_project_dependencies`, `sketch_callgraph`,
  `explore_codebase` — заготовки под будущий индекс.

## 3. Что не совсем работает / нужно чинить

| Проблема | Где | Чинить |
|----------|-----|--------|
| `read_file` рубит большие файлы | filesystem.py | offset/limit по строкам + `file_outline` (только символы) |
| `glob`/`grep` молча обрезают результат | filesystem.py | явный признак «обрезано, ещё N», пагинация, учёт `.gitignore` |
| callgraph: потолок 150 файлов, плоские рёбра | callgraph.py | tree-sitter, без потолка, индекс достижимости |
| recon — свободный текст, теряется на больших репо | graph.py | заменить на структурную Repo Map |
| `coverage: done` не отражает реальное покрытие | schema.py/graph.py | проценты: % файлов / entry points / sinks реально просмотрено |
| дедуп слабо схлопывает находки разных специалистов | schema.py | нормализовать путь к корню репо; fingerprint по пути+CWE+символу |
| многострочный PoC обрезается парсером markdown | schema.py | `with_structured_output` у специалистов вместо markdown |
| специалисты переотчитываются (19 находок на 1 файл) | prompts.py | триаж-режим + ужать промпты |
| валидатор перечитывает файлы на каждую находку (280 с/19) | validator.py | кэш чтений + Repo Map + батч низких severity |

## 4. Опора 1 — контекст кодовой базы (Repo Map)

Новый этап `index` (узел графа до recon) строит **Repo Map** —
структурированный, персистентный контекст репозитория:

- **Инвентаризация файлов**: путь, язык, размер, LoC, роль
  (source/test/config/vendored/generated), git-метаданные.
- **Attack surface / точки входа**: HTTP-роуты, CLI-команды, обработчики
  очередей, вебхуки, точки десериализации — детект по фреймворку
  (Flask/Django/FastAPI/Express/Spring/…).
- **Индекс символов**: функции/классы/методы с `file:line` — через
  **tree-sitter** (мультиязычно, детерминированно, быстро).
- **Граф вызовов и импортов**: настоящая достижимость (callers/callees),
  без потолка в 150 файлов.
- **Карта sink-ов**: места опасных операций (exec/eval/subprocess, сырой
  SQL, файловые операции, десериализация, криптопримитивы) —
  детерминированно через semgrep + grep-паттерны.
- **Зависимости + известные CVE** (переиспользуем `dependencies.py`/OSV).

Свойства Repo Map:

- **Персистентность**: кэшируется по `repo + git commit` (в каталоге
  `.agentsec/` цели или в session-БД). Строится один раз, переиспользуется.
- **Инкрементальность**: на следующем прогоне переиндексируются только
  изменённые файлы.
- **Запрашиваемость**: новые инструменты вместо слепого grep —
  `find_entry_points`, `find_sinks(kind)`, `who_calls(symbol)`,
  `callers_of`/`callees_of`, `symbol_lookup`, `repo_map_summary`.
  Это и есть «контекст»: специалист идёт сразу по пути
  attack surface → sink, а не перебирает файлы.

## 5. Опора 2 — масштабируемый прогон

**5a. Триаж-режим.** Сканеры + grep по sink-паттернам в фазе `index`
дают список **кандидатов** (`file:line`, правило, класс). Специалисты
**триажат кандидатов** (подтвердить / расширить / отклонить), а не ищут
с нуля. Объём работы привязан к числу кандидатов, а не к размеру репо.
Ценность LLM — рассуждение о достижимости/эксплуатируемости и поиск
того, что сканеры пропустили, в приоритетных файлах.

**5b. Шардирование fan-out.** Вместо «3 специалиста на весь репо» —
разбить репо на единицы анализа (по модулям/каталогам или кластерам
точек входа), ранжировать по приоритету attack surface, разложить
работу как (шард × класс). Динамический fan-out через LangGraph `Send`.
Каждая единица мелкая → влезает в контекст, ограничена по времени,
параллельна. `consolidate` сливает. Это даёт измеримое покрытие.

**5c. Бюджеты и приоритизация.** Бюджет на прогон: max файлов, max
LLM-вызовов, max токенов, wall-clock. Приоритет — файлы у точек входа,
sink-плотные, недавно изменённые. Бюджет исчерпан → честный
**измеренный** отчёт о покрытии (просмотрено X% файлов / Y% точек
входа), а не вводящее в заблуждение `done`.

**5d. Инкрементальный / diff-режим.** `--diff <base>` / PR-режим:
анализировать только изменённые файлы + их зону влияния
(callers/callees изменённых символов из графа). Это корректный скоуп
для CI-quality-gate и быстрый гейтинг на больших репо. Зону влияния
делает вычислимой граф вызовов из Repo Map.

**5e. Инструмент ретрива.** `search_code`: сначала структурный поиск
(tree-sitter / ast-grep запросы — точно и дёшево), семантический
(эмбеддинги) — опционально позже.

## 6. Решения

- **tree-sitter — принято.** Мультиязычный детерминированный парсинг;
  только `ast` Python недостаточно для полиглот-репо (есть в ТЗ).
- **Эмбеддинги/RAG — отложено (опционально).** Структурный индекс
  закрывает большую часть «контекста»; вектор-стор добавляет модель,
  стоимость и сложность. Решать после Фазы 2.
- **ast-grep — опционально** как движок структурных запросов; tree-sitter
  уже даёт основу, ast-grep — поверх при нехватке.
- **Кэш Repo Map** — каталог `.agentsec/` в цели, ключ — git commit;
  работает и без режима сессий.
- **DAST/сетевые инструменты — по-прежнему вне скоупа** (см.
  [orchestrator-plan.md](orchestrator-plan.md) §5): система — SAST
  quality gate по коду.

## 7. Фазы и зависимости

**Фаза 1 — фундамент Repo Map** (наивысшая ценность, разблокирует всё):
1. Зависимость `tree-sitter` + грамматики; пакет `agentsec/index/`.
2. Инвентаризация файлов + индекс символов + точки входа + sink-карта.
3. Персистентность по git-commit, инкрементальная переиндексация.
4. Узел `index` в графе до `recon`; Repo Map в состоянии.
5. Инструменты-запросы Repo Map; подключить к recon и специалистам.
6. `read_file` offset/limit + `file_outline`; явная обрезка glob/grep.

**Фаза 2 — триаж и масштаб** (зависит от Фазы 1):
1. Список кандидатов из сканеров + sink-grep; узел `triage`.
2. Специалисты переходят на потребление кандидатов.
3. Менеджер бюджета + приоритизация; честное покрытие в процентах.
4. Шардированный fan-out через `Send` (шард × класс), join на consolidate.

**Фаза 3 — инкрементальный / CI** (зависит от графа вызовов Фазы 1):
1. `--diff`/PR-режим: изменённые файлы + зона влияния по графу.
2. Переиспользование кэша Repo Map, инкрементальная переиндексация.

**Фаза 4 — качество** (слабо зависит от 1–3):
1. `with_structured_output` у специалистов — убрать round-trip через
   markdown (чинит обрезку PoC).
2. Масштабирование валидатора: батч низких severity, кэш чтений,
   Repo-map-aware.
3. Нормализация путей в `Finding` и дедупе.
4. (Опционально) семантический ретрив на эмбеддингах.

## 9. Метрика успеха

- Прогон на репо в сотни-тысячи файлов завершается в предсказуемое
  время (бюджет/таймауты), отчёт содержит **измеренное** покрытие.
- Повторный прогон того же репо переиспользует Repo Map (инкрементально).
- diff-режим на PR — единицы минут вместо десятков.
- Eval (`eval/`) переведён на структурные находки: precision/recall на
  больших таргетах, а не substring-match.
