# План улучшений: качество детекта на больших репо

Прогон Grafana показал: инфраструктура больших репо готова (индекс,
триаж, бюджеты, инкрементальность, приоритизация), но детект даёт ~0 —
regex-эвристика sink-ов на зрелом коде = почти сплошь ложные кандидаты,
дешёвая модель их отклоняет.

## Шаги

1. **Валидатор: устойчивый разбор JSON.** Прогон Grafana: 4 из 5 находок
   валидатор не распарсил (одинарные кавычки / markdown-обёртка) →
   остались `unverified`. Чинится `_extract_json`.

2. **Сканеры как источник кандидатов (ядро).** semgrep с курируемыми
   security-правилами даёт настоящие кандидаты с dataflow, не regex-шум.
   - semgrep гоняется в фазе `index` по файлам-кандидатам Repo Map
     (sink-и/точки входа) — не по всем 16k, поэтому быстро;
   - находки кладутся в таблицу `scanner_findings` индекса, кэш по
     git-commit (инкрементально);
   - триаж подмешивает их как кандидатов с высшим приоритетом
     (semgrep точнее нашей regex-эвристики);
   - gitleaks/osv — если установлены: gitleaks → secret-кандидаты,
     osv → находки напрямую (детерминированно, без LLM).

3. **Маппинг правило→класс.** semgrep `check_id` по ключевым словам →
   injection / secrets / authnz.

## Решения

- semgrep по файлам-кандидатам, а не по всему репо — иначе на 16k
  файлов он медленный; Repo Map уже сузил поверхность.
- Кэш находок сканера по commit — как и сам индекс.
- Деградация: нет semgrep/gitleaks/osv → пайплайн работает на
  regex-кандидатах, без падений.

## Отложено

- Прогон semgrep по всему репо (медленно) — пока только по кандидатам.
- Сильная модель для триажа — вне кода (вопрос `.env`).
- diff/PR-режим — отдельная фаза.
